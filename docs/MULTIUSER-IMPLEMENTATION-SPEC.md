# WatchTrack Multi-User Implementation Spec (v6.0)

**Purpose:** turn WatchTrack from single-user into a 5–20-person friends-and-family app. Hand-off document for Claude Code.

**Status:** locked design, ready to implement.

**Out of scope for v1:** cast/handoff (send-to-TV), per-user Plex servers, Google Play distribution. Designed for, but not built in, this release.

---

## 1. Decisions recap

| Area | Decision |
|---|---|
| Identity | Device ID + server-issued bearer token. No usernames or passwords. |
| Registration | Single-use magic invite links generated by admin (Lincoln). |
| Trakt auth | Pure OAuth device-code flow. One shared Trakt application registered by Lincoln. Each user signs into their own Trakt account. |
| Plex | Lincoln-only in v1. Data model designed for per-user Plex in future. |
| Two-way QR | First-time TV pairing in v1. Cast/handoff deferred to v2 (uses same SSE plumbing). |
| Account recovery | Both: admin-issued reconnect links AND DIY QR rescue from another paired device. No recovery phrases. |
| Admin UI | Lightweight view visible only to Lincoln's userId. |
| Legacy auth | `CONFIG.secret` retained as Lincoln-only backdoor. New users never use it. |
| Distribution | APK at `cinemath.app/install.apk` + GitHub releases. Google Play deferred. |
| Scale target | 5–20 users. Stays within Cloudflare free tier. |

## 2. Data model

### 2.1 New KV namespace: `USERS`

```
user:{userId}
  {
    displayName: "Dad",
    createdAt: 1747400000000,
    role: "user" | "admin",                  // exactly one admin
    traktTokens: { access, refresh, expiresAt, scope } | null,
    plex: { url, token } | null,             // Lincoln-only in v1
    settings: { ... },                       // per-user UI prefs
    lastSeen: 1747400000000,
  }

device:{deviceId}
  {
    userId: "uuid",
    kind: "phone" | "tablet" | "tv" | "desktop",
    name: "Lincoln's iPhone",                // editable in settings
    userAgent: "...",                        // captured at registration
    createdAt: 1747400000000,
    lastSeen: 1747400000000,
    authTokenHash: "sha256(...)",            // store hash, not plaintext
  }

invite:{inviteCode}
  {
    createdBy: "{adminUserId}",
    createdAt: 1747400000000,
    expiresAt: 1748004800000,                // default 7 days
    consumed: false,
    consumedBy: "{userId}" | null,
    plexEnabled: false,                      // only true on Lincoln's own invite
    suggestedDisplayName: "Dad" | null,      // optional pre-fill
  }

reconnect:{reconnectCode}
  {
    targetUserId: "uuid",
    createdAt: 1747400000000,
    expiresAt: 1748004800000,
    consumed: false,
  }

pair:{pairSessionId}
  {
    createdAt: 1747400000000,
    expiresAt: 1747400300000,                // 5 minutes
    consumed: false,
    fulfillment: null | { newDeviceId, authToken, userId },
  }

admin
  { userId: "{lincolnUserId}" }              // singleton record
```

### 2.2 Existing KV namespaces: changes

| Namespace | Change |
|---|---|
| `CONFIG` | Keep `secret` (Lincoln-only legacy auth). Plex credentials migrate into `user:{lincolnUserId}.plex`. |
| `SYNC_KV` | Re-key from `user:HASH` to `state:{userId}`. Existing data migrates to Lincoln's userId. |
| `ALERTS` | Already per-user via HASH; migrate to `state:{userId}` keying. Cron walks all users. |
| `D1_VIEWED` | Add `userId TEXT NOT NULL` column. Backfill all existing rows to Lincoln's userId. Index `(userId, watchedAt DESC)`. |
| `PROMOTIONS` | Add `userId` to record shape. Filter on read. |
| `METADATA` | Unchanged. Shared TMDB cache benefits all users. |
| `BACKUPS` (R2) | Re-organize as `backups/{userId}/{date}.json.gz`. Daily cron walks user list. |
| `EVENTS` | Plex-only; stays attached to Lincoln (or future per-user Plex via per-user webhook routing). |

### 2.3 Static data

`/data/*.json` curated catalogs — **unchanged, shared across all users**. These are the curated lists shipped with the app. Per-user state is the watched/rated overlay on top.

## 3. Auth model

### 3.1 Bearer token flow (new default)

- Every authenticated request to the worker includes `Authorization: Bearer {authToken}`.
- Worker hashes the token, looks up `device:{deviceId}` (deviceId is sent as a header `X-Device-Id` for fast key lookup), confirms hash matches, resolves `userId`, scopes the request.
- All `userId`-scoped reads/writes use this resolved userId. No client can spoof another user's data by changing a query param.
- Tokens are 256-bit random, generated at registration time, sent over HTTPS only.
- Token revocation: `device:{deviceId}` deletion immediately invalidates the token. Used by admin "revoke device" and reconnect flow.

### 3.2 Legacy `CONFIG.secret` (Lincoln only)

- If a request omits bearer but includes `?secret=` matching `CONFIG.secret`, worker resolves it to Lincoln's userId only.
- This preserves backward compat for Lincoln's existing devices and the Plex webhook receiver.
- New users never receive or use the secret.
- Any attempt to use the secret from a request that doesn't resolve to Lincoln's user → 401.

### 3.3 CORS

- Change `Access-Control-Allow-Origin: *` to a whitelist of:
  - `https://cinemath.app` (or whatever production domain is)
  - `https://bicyclecrasher.github.io`
  - TWA origin (`android-app://...`)
- Reject other origins.

### 3.4 Per-user rate limiting

- Token bucket in KV: `ratelimit:{userId}:{endpoint}`.
- Defaults:
  - `/chat` (Workers AI): 30 calls/hour/user.
  - `/metadata/lookup`: 200 calls/hour/user (TMDB cache absorbs most).
  - Other endpoints: 600 calls/hour/user (10/min average).
- 429 response on exceed, with `Retry-After` header.

## 4. Onboarding flow

### 4.1 Magic invite link

URL: `https://cinemath.app/?invite={inviteCode}`

Admin generates from admin view → POSTs `/admin/invites/create` → worker creates `invite:{inviteCode}` record → admin copies/shares URL.

Client behavior on load with `?invite=`:
1. Check localStorage for existing `deviceId` + `authToken`.
2. If present → existing user; ignore the invite, drop into normal app.
3. If absent → enter onboarding mode:
   - Generate fresh `deviceId` client-side (UUID v4).
   - POST `/register` with `{ invite, deviceId, deviceKind, userAgent }`.
   - Worker validates invite (exists, not expired, not consumed). On success:
     - Generate `userId` (UUID v4).
     - Generate 256-bit `authToken`.
     - Create `user:{userId}` with `role: "user"`, `displayName: invite.suggestedDisplayName || null`.
     - Create `device:{deviceId}` bound to userId.
     - Mark invite consumed.
     - Return `{ userId, authToken, displayName, needsDisplayName }`.
   - Client stores `{ deviceId, authToken, userId }` in localStorage.
   - Enter wizard.

### 4.2 Setup wizard

Step 1 — **Display name** (if `needsDisplayName`):
- Free text, 1–30 chars. PUT `/user/settings`.

Step 2 — **Connect Trakt** (Trakt device-code flow):
- Client POSTs directly to Trakt: `https://api.trakt.tv/oauth/device/code` with shared `client_id`.
- Receives `{ user_code, device_code, verification_url, interval, expires_in }`.
- UI displays:
  - Large monospace `user_code` (e.g. `7B3F-K9X2`), tap-to-copy.
  - Tappable button "Open trakt.tv/activate" (opens in new tab; deep-links to verification_url).
  - Status text: "Waiting for approval…" with subtle spinner.
- Client polls Trakt every `interval` seconds: POST `/oauth/device/token` with `device_code` and `client_id` only (no client_secret per device-code flow).
- On 200 with tokens → POST tokens to worker `/trakt/store` (bearer-authed). Worker encrypts at rest, writes to `user:{userId}.traktTokens`.
- On 400 with `expired_token` → reset to step 2 start.
- On user-cancel during polling → allow restart.

Step 3 — **Plex** (skipped unless `invite.plexEnabled === true`):
- Lincoln-only in v1. Just a placeholder for the existing Plex config UI, scoped to current user.

Step 4 — **Done**. Land on home. Library = static catalogs + per-user Trakt watchlist overlay.

### 4.3 Trakt application setup (one-time, Lincoln)

Pre-implementation:
1. Lincoln registers a Trakt application at https://trakt.tv/oauth/applications.
2. Name: "WatchTrack" (or "CinéMath"). Redirect URI: `urn:ietf:wg:oauth:2.0:oob` (sentinel for device flow).
3. Capture `client_id`. Store in worker env var `TRAKT_CLIENT_ID` (also bundled in client for the device-code endpoint — `client_id` is not a secret in OAuth device flow).
4. No client_secret needed for device-code flow.

## 5. Two-way QR pairing (v1)

### 5.1 Pair-session lifecycle

1. **TV** (blank, no device binding) opens app → "Pair this device" screen.
2. TV POSTs `/pair/begin` (unauthenticated). Worker:
   - Generates 256-bit `pairSessionId`.
   - Creates `pair:{pairSessionId}` record with 5-minute TTL.
   - Returns `{ pairSessionId, qrUrl: "https://cinemath.app/p/{pairSessionId}" }`.
3. TV renders the QR encoding `qrUrl`. Large, high contrast, with WatchTrack branding.
4. TV opens SSE stream to `/pair/wait?session={pairSessionId}`. Stream stays open until consumed or timeout.

5. **Phone** (already authenticated) scans QR.
   - QR resolves to `https://cinemath.app/p/{pairSessionId}` which the PWA / TWA intercepts via deep-link routing.
   - Phone detects this is a pair-session intent.
   - UI shows confirmation card: "Pair a new device with your account?" — includes target device's user-agent and IP (returned from `/pair/info?session=...`).
6. User taps "Confirm." Phone POSTs `/pair/confirm` with bearer auth + `pairSessionId`.
7. Worker:
   - Validates phone's bearer.
   - Validates pair session exists, not consumed, not expired.
   - Generates new `deviceId` + `authToken` for the TV.
   - Creates `device:{newDeviceId}` bound to the same `userId` as the phone, with `kind: "tv"`.
   - Writes `fulfillment` field on pair session.
   - Pushes `{ event: 'paired', userId, deviceId, authToken, displayName }` over the TV's SSE stream.
   - Marks pair session consumed.
8. TV's SSE handler:
   - Stores `{ deviceId, authToken, userId }` in localStorage.
   - Reloads as authenticated TV device.

### 5.2 Security details

- Pair session ID is 256-bit random, single-use.
- Confirmation requires the phone's bearer auth → can't be exploited by a stranger scanning the QR.
- 5-minute TTL prevents long-lived sessions sitting around.
- Optional v1.1: also display a 4-character confirmation code on both screens, require match — defends against shoulder-QR-scanning. Skip for v1.

### 5.3 SSE plumbing

- Cloudflare Workers supports SSE via streaming Response.
- Use existing `EventSource` browser API on client.
- TV reconnects with backoff on disconnect (10s, 30s, 60s, then 60s steady).
- Worker keeps SSE-connection-id → pairSessionId mapping in Durable Object or in-memory; on confirm, pushes via that channel.
- After v1 ships, this same plumbing extends to per-device cast channels (`/devices/{deviceId}/stream`) for v2.

## 6. Recovery flows

### 6.1 Admin reconnect link (primary)

- Admin opens admin view → finds user by display name → clicks "Issue reconnect link."
- Admin POSTs `/admin/reconnect/create` with `{ targetUserId, expiresIn: 86400 }`.
- Worker creates `reconnect:{reconnectCode}` record.
- Admin copies URL `https://cinemath.app/?reconnect={reconnectCode}` and sends to user.
- User taps on new device:
  - Client detects `?reconnect=`, no existing binding.
  - POST `/reconnect` with `{ reconnectCode, deviceId, deviceKind, userAgent }`.
  - Worker validates, generates new `device:{deviceId}` bound to existing `user:{targetUserId}`.
  - Optional: revoke all prior devices on this user (offered as choice on the new device: "Replacing my old phone" vs "Adding another device").
  - Returns `{ userId, authToken, displayName }`.
- User's Trakt tokens, sync state, settings — all preserved.

### 6.2 QR rescue from another paired device (DIY)

- User goes to Settings on their working device (phone) → "Add another device."
- Phone POSTs `/pair/begin/owned` (bearer-authed, indicates this is a same-user pairing, not a fresh registration).
- Phone displays QR. New device scans.
- Same pair-session flow as §5, except the new device just gets bound to the existing user — no new user created.

### 6.3 Total device loss

User has lost all devices simultaneously → admin reissue is the only path. Admin reconnect flow.

## 7. Admin view

Lives in the app at `/admin` route. Visible only to users where `user.role === "admin"`.

### 7.1 Users tab
- Table: Display name • # devices • Last seen • Has Trakt • Has Plex • Actions
- Actions per row: rename, revoke device (per-device sublist), revoke entire user, issue reconnect link.

### 7.2 Invites tab
- Generate new invite: display name suggestion (optional), expiry (default 7d), plexEnabled flag (default false).
- List pending invites with copy-link button.
- List consumed invites with user they became.

### 7.3 Activity log (optional, nice-to-have)
- Last 50 admin-relevant events: user registered, device added, Trakt reconnected, device revoked.
- Capped circular buffer in KV (`activity:0`–`activity:49`).

### 7.4 Bootstrap
- On first deploy post-migration, the migrated Lincoln user has `role: "admin"`. There can only be one admin. Set via a one-time worker env var `BOOTSTRAP_ADMIN_USER_ID` on first deploy.

## 8. Migration plan

Single migration runs once on worker startup if `USERS:admin` record doesn't exist.

1. Read existing `CONFIG.secret`. Confirm it exists.
2. Generate Lincoln's `userId` (UUID v4).
3. Create `user:{lincolnUserId}`:
   - `displayName: "Lincoln"` (or env var override).
   - `role: "admin"`.
   - `plex: { url: CONFIG.plex_url, token: CONFIG.plex_token }`.
   - `traktTokens`: extract from current storage location (TBD — Claude Code should verify where Trakt tokens currently live; likely localStorage on Lincoln's device, in which case the existing app needs a one-time POST to migrate them on first run).
4. Re-key `SYNC_KV`: scan keys matching `user:*`, copy to `state:{lincolnUserId}`, delete old.
5. Re-key `ALERTS` similarly.
6. Add `userId` column to D1_VIEWED, backfill all rows with `lincolnUserId`. Create index.
7. Update PROMOTIONS records to include `userId: lincolnUserId`.
8. Write `USERS:admin` singleton with `{ userId: lincolnUserId }`.
9. Worker logs `MIGRATION_COMPLETE` with userId. Subsequent boots skip.

Client migration (Lincoln's existing device on first launch post-deploy):
- App detects new `/migrate` endpoint returns 200 with `{ userId, authToken }`.
- POST `/migrate` with `{ oldSecret: CONFIG.secret, deviceId }`.
- Worker validates secret matches, returns Lincoln's userId + new authToken.
- Client stores in localStorage, switches all future requests to bearer.
- Old secret remains valid as fallback (per §3.2).

## 9. Distribution

### 9.1 APK build pipeline

- Extend existing `worker/build-twa.sh` (or wherever the TWA build lives) to produce a signed release APK.
- GitHub Actions job: on tag push (`v6.0.0+`), build APK, sign, attach to GitHub release.
- Host `cinemath.app/install.apk` as a 302 redirect to the latest GitHub release APK asset.

### 9.2 Install guide

- Create `bicyclecrasher.github.io/install.html` — single-page visual walkthrough:
  - Phone: "Tap your magic link → tap Install when prompted." Most novices won't need the APK path.
  - Android TV / Fire TV: 4-step screenshot guide — enable unknown sources, open TV browser, navigate to `cinemath.app/install.apk`, install.
- Link this page from every invite email/text template Lincoln uses.

### 9.3 TV browser fallback

- For TVs where APK install isn't an option (or for first-touch before installing), the magic invite URL works in any TV browser. Manifest's `display: standalone` makes the "Add to Home Screen" path produce a launcher.

### 9.4 Update mechanism

- PWA auto-updates via service worker (already in place).
- APK updates: the TWA wraps the PWA, so most updates happen automatically via PWA refresh. Native shell updates (rare) require manual reinstall or future Play Store distribution.

## 10. Implementation sequencing

Recommended order. Each phase ends in a deployable, working state.

### Phase 1 — Backend foundation (no user-facing changes)
1. Add `USERS` KV namespace. Update `wrangler.toml`.
2. Implement migration code path; deploy; verify it ran once and Lincoln's user record exists.
3. Implement bearer-auth middleware. Legacy secret still works for Lincoln.
4. Migrate Lincoln's client to use bearer (`/migrate` endpoint flow). All API calls now bearer-authed for Lincoln.
5. Lock CORS to whitelisted origins.
6. Implement per-user rate limiting.

### Phase 2 — Invite + registration
7. `/admin/invites/create`, `/admin/invites/list`, `/admin/invites/revoke`.
8. `/register` endpoint.
9. Admin view UI (invites tab only, minimum viable).
10. Onboarding wizard: display name step, Trakt device-code step.
11. Test end-to-end with a throwaway invite on a second browser.

### Phase 3 — Trakt scope-per-user
12. Existing Trakt calls in the client refactored to use the current user's token from the worker, not a global one.
13. Worker encrypts Trakt tokens at rest (use Web Crypto with a key from env var).
14. Token refresh logic per user.

### Phase 4 — Admin tooling completion
15. Users tab in admin view.
16. Reconnect link generation and `/reconnect` endpoint.
17. Per-device revoke + rename.

### Phase 5 — QR pairing
18. `/pair/begin`, `/pair/info`, `/pair/confirm`, `/pair/wait` (SSE).
19. TV pairing UI (blank-device entry screen with QR display + SSE listener).
20. Phone scan handling: deep-link route for `/p/{pairSessionId}`, confirmation card.
21. Settings → "Add another device" (uses `/pair/begin/owned` flow).

### Phase 6 — Distribution
22. GitHub Actions APK build + release pipeline.
23. `cinemath.app/install.apk` redirect.
24. Visual install guide on `bicyclecrasher.github.io/install.html`.

### Phase 7 — Polish
25. Per-user backup folders in R2; update daily cron walker.
26. Webhook URL restructure to `/webhook/v2/{userWebhookId}` (Lincoln only for now; future-proofs per-user Plex).
27. Activity log in admin view.

### Out of scope (v2)
- Send-to-TV cast/handoff (builds on §5 SSE plumbing).
- Per-user Plex.
- Google Play Store publishing.
- Optional 4-char confirmation code on QR pairing.
